# 4.2 Solution Overview Figure（解决方案总览图）

## 核心目标

Solution Overview Figure 是论文 Methodology 章节的"灵魂"。它的使命是让读者（尤其是那些跳过大量文字直接看图的审稿人）迅速掌握你的整体系统架构、核心模块以及它们之间的数据流转关系。

这张图通常出现在 Methodology 章节的开头（如 Section 3 的第一段之后），是读者理解你整个方法的"地图"。一个好的 Overview Figure 能让读者在阅读后续技术细节时始终保持全局视角，知道当前讨论的模块在整个系统中的位置。

## 常见的设计范式

### 范式一：Pipeline / 流水线图

这是最常见的范式。适用于你的方法是一个多阶段的处理流程，从左到右（或从上到下）依次展示：输入 → 阶段1 → 阶段2 → ... → 输出。每个阶段用一个大方框表示，大方框内部可以包含子模块的更多细节。

**案例参考**：AFlow（ICLR 2025）的 Figure 3 采用了清晰的三栏布局：Search Space（左栏，展示搜索空间的定义，包括 Node、Operator、Edge 的代码表示）→ Search via AFLOW（中栏，展示 MCTS 搜索过程的四个步骤：Selection、Expansion、Evaluation、Backpropagation）→ Search Result（右栏，展示搜索得到的不同任务的最优工作流）。整个图的逻辑极为顺畅，读者从左到右一看就懂。

![AFlow Figure 3：Pipeline / 流水线图示例](../../assets/images/aflow_solution_overview.png)

### 范式二：系统架构图

适用于你的方法是一个包含多个交互组件的系统。用大方框表示系统边界，内部划分出若干子模块，用箭头表示模块之间的数据流和控制流。

**案例参考**：Alpha-SQL（ICML 2025）的 Figure 3 展示了 MCTS 框架的整体架构，包括 LLM-as-Action-Model、搜索树的展开过程、以及自监督奖励函数的评估反馈环路。各模块之间的数据流向清晰明了。

![Alpha-SQL Figure 3：系统架构图示例](../../assets/images/alpha_sql_solution_overview.png)

### 范式三：多层次 / 多视角图（线下-线上模块图）

适用于你的方法涉及多个抽象层次（如从宏观到微观）或多个视角（如训练阶段和推理阶段）。可以使用上下分层或左右分栏来展示不同层次/视角。

**案例参考**：LEAD（VLDB 2026）的 Figure 3 采用上下结构，展示了整体框架 Offline 和 Online 分别做了什么，以及 Online 的输出是如何与 Training 模块交互的。上下结构一般用于方法由 Offline 和 Online 两个大模块组成的情况，左右分栏则常用于输入-处理-输出的线性流程。

![LEAD Figure 3：多层次 / 多视角图示例](../../assets/images/lead_solution_overview.png)

## 设计原则

- **模块化**：每个核心组件用一个方框表示，方框内部可以进一步细分。
- **数据流清晰**：用箭头明确标注数据的流向，避免交叉和混乱。
- **层次分明**：通过颜色、大小和位置来区分不同层级的组件。
- **与正文对应**：图中每个模块的名称必须与正文中的章节标题一致。
- **输入输出明确**：最基本的要求是让读者知道方法的【输入】是什么、【输出】是什么，必须在图中清楚表达。

> **实用建议**：不管用什么工具，画 Overview Figure 之前，强烈建议先在纸上手绘一个草图，与导师讨论确认布局和信息层次后再上机绘制。这能避免大量的返工。也可以参考第五章 Vibe Research，了解如何用 AI 辅助绘图。
